## 📋 工作流与并行调度

### 1. 任务分析与调度
- **依赖拆解**：识别任务依赖图，区分「可并行节点」与「串行节点」。
- **并行执行**：将无前置依赖、无写冲突的子任务打包，通过 `multi-agent` 同时下发。信息收集阶段应高度并行，分析阶段汇总后执行。
- **串行降级**：对于存在强依赖链（A→B→C）或同一文件多处复杂修改的任务，按原有模式顺序执行，不强行并行化。

### 2. 递归迭代与交付
- **结果整合**：等待并行任务返回，校验一致性，处理异常后整合为阶段性结果，递归下一轮。
- **验证交付**：任何修改必须通过自动化测试验证，不得标记未经验证的任务为完成。

## 🛠️ 工程门禁与质量标准

### 🚧 强制工程门禁
- **本地统一门禁**：执行 `npm run check:all` 必须全绿。
- **提交前校验 (pre-commit)**：必须依次通过 `lint`、`typecheck`、`test:related`。
- **Check 链路**：必须包含并按顺序执行 `lint` -> `typecheck` -> `test:coverage` -> `build` -> `check:rust`。

### 🏗️ 架构与前端规范
- **设计原则**：遵循 SOLID、DRY、关注点分离。复用已有 composable/service/store，禁止在 `App.vue` 堆砌逻辑。
- **语义与交互**：优先语义化 HTML（`main/section/nav/button` 等），禁止为样式牺牲语义。新增 UI 必须同步键盘可达性（focus、Esc 关闭等）。
- **清理原则**：修改功能时不保留旧的兼容性代码，坚决删除无用代码。
- 考虑时间/空间复杂度，并在相关情况下优化高 I/O 或内存使用。
- 明确处理特殊情况；不要隐藏失败。

### 📘 TypeScript 严格模式
- **禁止 Any**：严禁使用 `any`。确有必要时优先使用 `unknown` + 类型收窄。
- **直面报错**：禁止忽略类型错误（不允许“先关掉再说”）。

### 代码指标（硬性限制）
- 函数：≤ 50 行（不含空行），若超出，须立即提取辅助函数（Helper Function）。
- 文件：≤ 300 行，若超出，须按职责拆分文件。
- 嵌套：深度 ≤ 3 层，使用卫语句（Guard Clauses）或早期返回来展平逻辑。
- 参数：位置参数 ≤ 3 个，超过此数量请使用配置对象（Config/Options Object）。
- 复杂度：循环复杂度 ≤ 10，若分支过多，须进行逻辑解耦。
- 魔术值：零容忍。所有数值须提取为具名常量（例如：使用 MAX_RETRIES = 3，而非直接使用数字 3）。

> **核心不可变原则**：所有思考、分析、解释和回答必须使用**简体中文**。

### 💬 语言与结构
- **友好自然**：像专业朋友对话，开篇一句话概括核心思路，多用短句。
- **视觉边界**：标题必须独占一行，使用 `**粗体**` 且缀有 Emoji，**前后各空一行**。不同信息块之间用 2 个空行分隔。

### 🧩 代码与技术展示
- **聚焦核心**：使用带语言标识的 Markdown 代码块，省略无关导入，用 `+` / `-` 标注差异。提及文件时直接输出文件名（如 `App.vue`），忌用全路径。
- **图示规范**：仅在表达结构、流程、关系时使用简明的 ASCII 图示（限制 20 行内，配文字说明）。优先使用列表，仅在严格对齐时使用表格。

### 📝 文档同步规范
- **公开文档**：功能/行为改动后必须同步更新仓库公开文档（README、CHANGELOG 等）。内部笔记保留在本地。
- **双语入口**：影响开源用户使用入口（安装、命令、配置）的改动，必须同步更新 `README.md` 与 `README.zh-CN.md`。

### 3.fast-context - 代码检索

#### 核心原则
**任何需要理解代码上下文、探索性搜索、或自然语言定位代码的场景，优先使用 `mcp__fast-context__fast_context_search`**

#### 使用场景

##### 1️⃣ 必须用 fast_context_search
- 探索性搜索（不确定代码在哪个文件/目录）
- 用自然语言描述要找的逻辑（如"XX部署流程"、"XX事件处理"）
- 理解业务逻辑和调用链路
- 跨模块、跨层级查询（如从 router 追到 service 到 model）
- 新任务开始前的代码调研和架构理解
- 中文语义搜索（工具支持中英文双语查询）

##### 2️⃣ 根据需求选择工具
- **语义搜索 / 不确定位置** → `fast_context_search`（返回文件+行号范围+grep关键词建议）
- **精确关键词搜索** → Grep
- **已知文件路径，查看内容** → Read
- **按文件名模式查找** → Glob
- **编辑已有文件** → Edit

##### 3️⃣ fast_context_search 参数调优
- `tree_depth=1, max_turns=1` — 快速粗查，适合小项目或初步定位
- `tree_depth=3, max_turns=3`（默认）— 平衡精度与速度，适合大多数场景
- `max_turns=5` — 深度搜索，适合复杂调用链追踪
- `project_path` — 指定搜索的项目根目录，默认为当前工作目录
**CRITICAL：回答任何代码相关问题前，必须先调用搜索工具定位代码。禁止猜测或凭经验假设代码位置。**

#### 工具优先级

1. **已知函数名/类名** → `Grep`
2. **理解业务逻辑 / 探索代码 / 查找实现** → `Augment search_context`（返回完整代码片段，语义理解强）
3. **Augment 结果不足** → `Fast-Context`（返回文件路径+行号，附带 grep 关键词建议）
4. **Fast-Context 返回 grep keywords** → 立即用 `Grep` 二次精确搜索
5. **仍未找到** → 组合 Glob + Read + Grep


#### 禁止行为

- ❌ 猜测代码位置（"应该在 service/firmware 里"）
- ❌ 跳过搜索直接回答（"根据框架惯例，应该是..."）
- ❌ 遇到搜索就启动 Task/Explore 子代理（Augment + Fast-Context + Grep 组合优先）

#### 子代理使用条件

仅当需要读取 10+ 文件交叉比对、或多轮搜索会撑爆上下文时，才启动子代理。
