# ZapCmd 功能完善与改进建议

> **创建日期**：2026-02-28
> **状态**：讨论中
> **说明**：基于项目代码全面审查后的改进建议，按优先级排列

---

## 一、高优先级（近期迭代，v1.x）

### 1.1 命令历史与使用频率

**问题**：当前搜索排序完全基于文本匹配得分，不考虑用户行为。用户最常用的命令可能排在下面。

**方案**：
- 记录每条命令的执行次数和最后执行时间
- 存储到 `~/.zapcmd/stats.json`（模板文件已预留：`assets/runtime_templates/stats.json`）
- 搜索排序中加入频率权重：`finalScore = matchScore + frequencyBonus`
- 频率衰减：30 天未使用的命令权重归零，避免历史数据永久主导排序

**改动范围**：
- Rust 新增 `stats_catalog.rs`（读写 stats.json）
- `useLauncherSearch.ts` 扩展评分函数
- 约 3 个文件改动 + 1 个新增

**价值**：用越多越顺手，是命令启动器的核心体验。

---

### 1.2 命令收藏/置顶

**问题**：209 条命令中，用户可能只常用 10-20 条。没有快速访问入口。

**方案**：
- 支持标记命令为"收藏"
- 搜索框为空时显示收藏列表（而非当前的空白）
- 收藏存储到 settings 的 `favoriteCommandIds: string[]`

**改动范围**：
- `settingsStore.ts` 新增 favorites 字段
- `useLauncherSearch.ts` 空查询时返回 favorites
- UI 交互：搜索结果项增加收藏操作（如 Ctrl+D 或右键菜单）

**价值**：空搜索框变废为宝，提供即时访问最重要的命令。

---

### 1.3 命令执行结果反馈

**问题**：当前执行命令后，ZapCmd 窗口隐藏，用户需要切换到终端查看结果。无法知道命令是否成功。

**方案**（轻量级，不替代终端）：
- 执行后在系统通知区弹出执行状态通知（成功/失败）
- 使用 Tauri notification API
- 可在 Settings 中开关此功能

**改动范围**：
- `src-tauri/Cargo.toml` 添加 `tauri-plugin-notification`
- `commandExecutor.ts` 执行后发送通知
- `settingsStore.ts` 新增通知开关

**价值**：闭合反馈循环，用户不需要在窗口间切换确认。

---

### 1.4 用户命令热重载

**问题**：当前用户命令 JSON 修改后需要重启应用才能生效（README 已标注为 Known Limitation）。

**方案**：
- 使用 Rust 的 `notify` crate 监听 `~/.zapcmd/commands/` 目录变化
- 文件变化时通过 Tauri event 通知前端
- 前端收到事件后重新加载命令目录

**改动范围**：
- `src-tauri/Cargo.toml` 添加 `notify` 依赖
- `src-tauri/src/command_catalog.rs` 新增文件监听
- `src/composables/launcher/useCommandCatalog.ts` 监听重载事件

**价值**：消除 Known Limitation #2，改善用户自定义命令的开发体验。

---

### 1.5 命令参数历史记忆

**问题**：有参数的命令（如 `ssh {{host}}`），每次执行都要重新输入参数值。

**方案**：
- 记录每条命令最近使用的参数值（最多 5 条历史）
- 参数输入框支持下拉选择历史值
- 存储到 `~/.zapcmd/arg-history.json`

**改动范围**：
- 新增 `argHistoryStore` 或合并到 stats
- `LauncherParamOverlay.vue` 参数输入框添加历史下拉

**价值**：高频带参命令的效率大幅提升。

## 二、中优先级（稳定后迭代，v1.x - v2.0）

### 2.1 插件系统 / 命令包管理

**问题**：当前用户命令需要手动编写 JSON 放到目录。没有命令分享生态。

**方案**：
- 支持通过 URL 或 GitHub repo 安装命令包
- 命令：`zapcmd install https://github.com/xxx/zapcmd-docker-pack`
- 或 Settings 中提供安装入口
- 命令包 = 一个包含 `commands/*.json` 的 Git repo

**价值**：建立社区生态，降低用户创建命令的门槛。

---

### 2.2 命令变量系统

**问题**：当前参数系统是简单的文本替换。缺少全局变量（如项目路径、用户名等）。

**方案**：
- 支持全局变量定义：`~/.zapcmd/variables.json`
- 命令模板中可引用：`cd {{$PROJECT_ROOT}} && npm run build`
- 以 `$` 前缀区分全局变量和命令参数

**价值**：减少重复输入，一处定义多处使用。

---

### 2.3 命令预览增强

**问题**：当前 preview 是静态文本。带参数的命令在参数填写前无法看到最终命令。

**方案**：
- 参数填写时实时预览最终命令
- 带高亮显示已替换的参数部分

**价值**：所见即所得，减少参数填写错误。

---

### 2.4 多终端同时执行

**问题**：批量队列目前是顺序在一个终端窗口执行。有些场景需要并行。

**方案**：
- 暂存区支持标记"并行执行"模式
- 并行模式下每条命令在独立终端窗口执行

**价值**：适合启动多个服务的场景（如同时启动 frontend + backend + database）。

---

### 2.5 Snippet / 剪贴板模式

**问题**：有些命令不需要在终端执行，用户只想复制命令到剪贴板。

**方案**：
- 支持 Ctrl+C 复制当前选中命令
- 支持 Ctrl+Shift+C 复制渲染后的命令（参数替换后）
- 系统通知提示已复制

**价值**：覆盖"只想复制命令"的使用场景。

---

## 三、低优先级（长期规划，v2.0+）

### 3.1 E2E 自动化测试

**问题**：当前只有单元测试，没有端到端测试。README 已标注为 Roadmap。

**方案**：
- 使用 Playwright + Tauri test driver
- 覆盖核心流程：搜索 → 参数 → 执行 → 暂存 → combo

**价值**：防止回归，保障发版质量。

---

### 3.2 主题系统

**问题**：当前只有一套 UI 主题（`listary` 风格）。用户无法自定义外观。

**方案**：
- 支持 Dark / Light / 自定义主题
- 基于 CSS 变量系统切换
- Settings → Appearance 中选择主题

**价值**：满足个性化需求。

---

### 3.3 工作区 / 项目感知

**问题**：当前命令是全局的，不区分项目上下文。

**方案**：
- 支持"工作区"概念：不同项目目录使用不同命令集
- 检测当前活动窗口的项目路径
- 自动切换命令上下文

**价值**：针对不同项目提供精准命令推荐。

---

### 3.4 远程命令执行

**问题**：当前只支持本地终端执行。

**方案**：
- 支持 SSH 远程执行
- 在 Settings 中配置远程主机
- 命令执行时可选择目标主机

**价值**：运维场景的刚需。

---

### 3.5 命令脚本编辑器

**问题**：用户命令需要手动编写 JSON，门槛较高。

**方案**：
- 在 Settings 或独立窗口提供可视化命令编辑器
- 表单化创建命令（名称、模板、参数、类别等）
- 生成 JSON 自动保存到 `~/.zapcmd/commands/`

**价值**：大幅降低用户创建命令的门槛。

---

## 四、非功能性改进

### 4.1 性能监控

- 启动时间度量（从启动到可交互）
- 搜索延迟度量（输入到结果展示）
- 内存使用跟踪

### 4.2 错误上报（Opt-in）

- 可选的匿名错误上报
- 帮助发现和修复 crash / 异常

### 4.3 无障碍增强

- 屏幕阅读器完整支持（当前部分 aria 已实现）
- 高对比度模式
- 键盘导航完整覆盖率验证

### 4.4 国际化扩展

- 当前 zh-CN + en-US
- 可考虑增加：ja（日语）、ko（韩语）、de（德语）等
- 社区贡献翻译机制

## 五、建议迭代路线图

```
v1.0.0  (当前) — 首个开源版
v1.1.0          — Custom Combo 组合命令
v1.2.0          — 命令历史 + 使用频率排序
v1.3.0          — 收藏/置顶 + 空搜索显示收藏
v1.4.0          — 用户命令热重载 + 参数历史记忆
```

每个小版本保持 **1-2 个功能点**，确保小步可验证交付，与 AGENTS.md 的工程原则一致。
