# Settings 顶部层叠 / 快捷键自适应 / 滚动命中区精修设计文档

日期: 2026-03-19
方案: C — Strong Layered Header（更强层叠感）
状态: 设计已确认，待进入 implementation plan

---

## 背景

当前 Settings 稳定版已经完成原生标题栏 + 应用层 topbar 收口，但用户在实际使用中仍然指出 3 个明确问题：

1. 顶部导航仍然有“外层卡片包裹一层、里面再放 Tab”的壳体感，不够像 `docs/bug_img/Raycast.png` 的头部层次。
2. Hotkeys 页面右侧快捷键录制器仍然偏向固定宽输入框，视觉上过长、过重。
3. 内容区滚动条仍然可见，且在内容 `max-width` 之外的左右留白区域，鼠标滚轮不一定能稳定驱动中间内容滚动。

本轮目标不是重做 Settings，也不是修改任何设置项逻辑，而是在现有稳定版架构上完成一轮“头部层叠感强化 + 热键录制器去输入框感 + 全宽滚动命中区”的定向精修。

---

## 已确认约束

用户已明确确认以下决策：

- 顶部导航采用 **更强层叠感**，不是极简线性方案，也不是当前带外层壳体的方案。
- `SSegmentNav` 外层卡片壳体需要移除，但当前激活项仍保留 **轻胶囊** 激活态。
- Hotkeys 录制器改为 **每个字段按内容独立自适应**，不做整组等宽。
- 滚动条只隐藏视觉表现，**真实滚动能力保留**：滚轮、触控板、键盘滚动都必须正常。
- 即使内容区内部继续保持 `max-width` 居中布局，**鼠标悬浮在左右留白区域时**，滚轮也应能驱动中间内容一起滚动。
- 本轮只改 Settings UI 结构和样式，不改设置项数据结构、保存逻辑、热键录制逻辑。

---

## 方案对比

### A. 极简线性

头部仅保留一条分割线，Tab 只做文字高亮或下划线；Hotkeys 与滚动问题分别做最小修补。

优点：

- 改动最小
- 风险最低

缺点：

- 头部层次偏弱，不符合用户对 Raycast 风格“上下层”的预期
- 容易做成过薄的网页导航，而不是桌面设置窗口头部

### B. Raycast 式双层感

去掉外层壳体，由 topbar 自己通过上下轻分色 + 分界线形成双层感；当前 Tab 保留轻胶囊。

优点：

- 已足够接近参考图
- 视觉平衡稳妥

缺点：

- 仍然偏“克制稳态”，层次冲击力略弱

### C. 更强层叠感（采用）

保留“去外壳 + 激活轻胶囊”的基础，但进一步强化头部作为“标题栏下第二层工具条”的结构感：上层更深、下层略亮、两层之间有明确横向分界，头部底边再与内容区形成第二条更细的分界。

优点：

- 最贴近用户指向的“像 Raycast，但不要卡片包裹”
- 更适合桌面设置窗口，而不是普通网页 tab 导航

缺点：

- 比 B 更依赖细节控制，若边框/阴影过重会重新做出卡片感

结论：采用 **C. 更强层叠感**。

## 详细设计

## 一、顶部导航结构

### 设计

- 保留 `SettingsWindow.vue` 现有的“topbar 在内容区之上”的物理结构，不回退到自定义标题栏方案。
- 删除 `.settings-window-topbar__nav-shell` 作为独立视觉壳体的职责，不再让导航簇包裹在一张圆角卡片里。
- `topbar` 自身变成头部层级容器：
  - 上半区视觉上贴近原生标题栏，颜色更深；
  - 下半区作为 Tab 承载层，颜色略亮；
  - 中间通过一条明确横向分界线表现“上下两层”；
  - 头部最底部再给内容区一条更细的边界线，形成“头部层 / 内容层”断开。
- `SSegmentNav` 继续居中，但只负责 Tab 本身，不再负责提供外层圆角壳体。
- `commands` 路由仍保留更宽内容宽度，但不再通过“更宽 nav-shell”表达，而是通过内容层宽度变体表达。

### Tab 激活态

- 默认 Tab 保持透明底，仅通过文字与图标层级体现可点击性。
- 当前激活项保留 **轻胶囊**：
  - 轻背景；
  - 轻边框；
  - 轻投影；
  - 不做厚重卡片阴影。
- Hover 态可略微提亮，但不能使导航重新出现“多张小卡片并排”的观感。

### 目的

- 去掉“卡片里再放 tab”的双重包裹感。
- 让头部更像原生窗口标题栏下的一层工具带。
- 维持键盘可达性与现有语义化 `tablist/tab` 结构不变。

---

## 二、Hotkeys 区与录制器自适应

### 设计

- 保留现有三个分组：
  - 全局快捷键
  - 搜索区快捷键
  - 执行流快捷键
- 分组标题继续放在组外部，作为 muted 小标题存在。
- 分组内部仍允许保留必要边框与行分隔，但整体“卡片外框存在感”进一步减弱，重点从整块轮廓转向每一行内容。

### 行布局

- 每一行继续采用“左标签 / 右录制器”的信息结构。
- 左侧标签区允许更自然地承接中短文案，不为给录制器让宽而过度压缩。
- 右侧录制器区不再固定 `280px`，而是根据内容宽度自适应。

### 录制器宽度策略

- 每个字段独立按内容宽度伸缩：
  - `Tab`、`Esc`、`Enter` 等短组合键会明显更短；
  - `Ctrl+Backspace`、`Alt+ArrowUp` 等长组合键会自然变宽。
- 同时保留两个边界：
  - **最小点击宽度**：避免空值或超短值过小，不利于点击和录制；
  - **合理最大宽度**：避免超长组合键把整行挤坏。
- 空值态、录制态、冲突态都必须维持稳定宽度逻辑，不能出现录制中突然塌缩或冲突提示把主按钮撑坏的情况。

### 录制器视觉方向

- 录制器整体更像“内容驱动的键帽容器”，而不是固定宽输入框。
- 内部 `kbd` token 风格保留，继续强调快捷键语义。
- 录制器本体与 token 之间仍需保留足够点击区域和焦点轮廓。
- 冲突提示继续挂在当前字段下方，作为该字段的局部反馈，不回退成全局 banner。

### 响应式退化

- 窄宽度下允许从左右布局退化为上下布局。
- 退化后顺序仍然是：标签在上，录制器在下，冲突提示紧跟录制器。
- 退化仅用于避免挤压，不应成为桌面默认布局。

### 目的

- 弱化“很多固定宽输入框排排站”的表单感。
- 强化桌面设置项中快捷键控件应有的轻量、精确、按内容收缩的观感。

## 三、滚动结构、隐藏滚动条与全宽命中区

### 问题根因

当前 `.settings-content` 同时承担了：

- 纵向滚动容器；
- `max-width` 居中阅读宽度；
- 内容本体容器。

当滚动容器本身被限制在中间阅读宽度时，左右留白区域不属于真实滚动容器命中区，鼠标悬浮在留白上滚轮可能没有效果。

### 设计

滚动结构拆为两层：

- **外层全宽滚动宿主**：负责承接整块内容区域的纵向滚动。
- **内层限宽内容容器**：只负责 `max-width`、居中、页面内排版。

也就是说：

- 视觉上仍然是“内容居中、有宽度上限”；
- 交互上则是“从左到右整条内容带都可以接收滚轮”。

### 滚动行为

- 顶部 topbar 固定，不参与滚动。
- 下方只保留一个主滚动宿主，不引入第二个纵向滚动链。
- 滚轮、触控板惯性滚动、方向键、Page Up/Page Down、Home/End 都继续生效。
- 不引入 JS 伪滚动，不做 wheel 事件劫持到自定义计算，优先使用正确的容器结构 + CSS 完成。

### 滚动条可见性

- 视觉滚动条完全隐藏：
  - Firefox 侧隐藏 `scrollbar-width`；
  - WebKit 侧隐藏 `::-webkit-scrollbar`。
- 但 `overflow-y` 仍应保持可滚动语义，不可用 `hidden` 伪装“隐藏滚动条”。

### 路由宽度规则

- 普通页面内层内容保持 `max-width: 720px` 的阅读宽度。
- `Commands` 页面内层内容继续放宽到既有的大宽度上限。
- 无论是普通页面还是 `Commands` 页面，滚动宿主都保持全宽，因此左右留白区滚轮行为一致。

### 目的

- 隐藏滚动条而不损失真实可滚动能力。
- 消除“鼠标必须对准中间内容本体才能滚”的死区。
- 为后续更多页面保留统一的滚动结构基线。

---

## 四、无障碍与交互稳定性

### 保持不变

- `SSegmentNav` 继续使用语义化 `tablist/tab`。
- 热键录制器继续使用按钮语义和现有键盘录制流程。
- 内容区仍是单主滚动区，不制造焦点陷阱或隐藏滚动区。

### 需要确保

- 焦点可视化不能因为录制器宽度自适应而消失或被裁切。
- 当前激活 Tab 的视觉强化不能替代 `aria-selected` 等语义状态。
- 隐藏滚动条后，键盘用户仍可清楚使用方向键/分页键滚动内容。

## 非目标

- 不改 Settings 数据结构、持久化模型或即时保存机制。
- 不修改热键录制逻辑、快捷键规范化逻辑或冲突检测逻辑。
- 不重做其他页面的信息架构。
- 不重新引入自定义标题栏或窗口控制按钮。

## 风险与注意点

- 头部层叠感如果边框、阴影或底色处理过重，会重新做回“卡片包裹感”。
- 录制器按内容自适应后，空值态与录制态若没有最小宽度，会出现点击热区过小的问题。
- 滚动结构拆层时必须保证只有一个纵向滚动链，避免 `outer auto + inner auto` 双滚动。
- 隐藏滚动条后，仍需确保用户能通过视觉节奏自然感知“还有内容可向下滚动”。

---

## 验收标准

1. 顶部导航不再存在独立的外层卡片壳体，但头部能明显看出上下两层关系。
2. 当前激活 Tab 保留轻胶囊态，默认项不再呈现并排小卡片感。
3. Hotkeys 录制器不再固定宽度，而是每个字段按内容独立自适应，并保留最小点击宽度与合理最大宽度。
4. Hotkeys 页面在桌面常规宽度下不再出现“固定宽输入框过长”的视觉问题。
5. 内容区滚动条视觉上完全隐藏，但滚轮、触控板、键盘滚动全部正常。
6. 内容继续保持居中与宽度上限，但鼠标悬浮在左右留白区域时，滚轮也能驱动内容滚动。
7. `Commands` 路由的放宽宽度规则仍然成立，且与新的全宽滚动宿主兼容。
8. 不引入业务逻辑回归；Settings 相关 focused tests 与全量检查保持通过。

## 增量修订（2026-03-19｜顶部 Tab 视觉对齐 Raycast）

在用户对首版实现做实机复核后，补充确认两条更具体的视觉约束：

1. 顶部导航中部横线属于多余噪音，需移除，只保留底部分隔线作为头部与内容区的边界。
2. Tab 的 hover / active 形态不再使用胶囊，而是向 `docs/bug_img/Raycast.png` 靠拢为更克制的圆角矩形；同时加强两组距离感：
   - Tab 与 Tab 之间的水平间距更明确；
   - Tab 组与底部分隔线之间保留稳定的垂直留白。

这次修订不改变此前已确认的业务边界，仍然只处理 Settings 顶部导航的视觉语言，不触碰 store、持久化或快捷键逻辑。

## 增量修订（2026-03-19｜顶部 Tab 默认态去框）

在用户第二轮实机复测后，进一步收紧顶部 Tab 的状态表达规则：

1. 默认态不应出现任何“按钮框”观感，基础态只保留点击热区，不通过可见边框、描边或阴影表达存在感。
2. hover / active 只通过背景色加深与文字提亮表达状态，不再额外使用可见边框或投影强化。

这条修订优先级高于前述“矩形化”描述中的具体视觉手法，用于约束默认态必须足够安静、状态态才出现明确强调。
