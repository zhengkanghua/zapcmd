# Settings 滚动与控件精修设计文档

日期: 2026-03-18
方案: B — 结构精修（在现有稳定版 Settings 架构上收口视觉与布局缺陷）
状态: 设计已确认，待进入 implementation plan

---

## 背景

`2026-03-18-settings-ui-refinement-design.md` 已完成 Settings 稳定版精修，但当前 Windows 实机截图仍暴露 4 类问题：

1. 内容较多时，Settings 内容区没有出现预期滚动条，底部信息被截断。
2. Hotkeys 页面纵向密度过高，部分行在较窄宽度下容易挤压，录制器看起来过长，卡片之间缺少呼吸感。
3. General 页面中的自定义下拉框视觉偏粗糙、尺寸偏大，与当前深色桌面风格不匹配。
4. 默认终端下拉框把路径作为 option description 展示，信息层级错误，长路径破坏了选择器可读性。

本轮目标不是重做 Settings，也不修改任何设置项业务逻辑，而是在现有架构下完成一轮“结构参考 Gemini、视觉保留 ZapCmd”的定向精修。

---

## 已确认约束

用户已明确确认以下约束：

- 只修 Settings 的视觉布局、信息呈现与控件样式，不改业务逻辑与交互语义。
- 滚动策略采用“整个 Settings 内容区滚动，顶部 Tab 固定不动”。
- 默认终端下拉框只显示终端名称，路径保留在下方独立只读行。
- 视觉方向采用“结构参考 `docs/bug_img/gemini.png`，配色与组件气质保留 ZapCmd 当前深色风格”。
- 实施方向采用“结构精修”，不走大改版。

---

## 设计目标

### 1. 修正滚动失效

建立清晰的高度约束链，让 Settings 内容在超出窗口可视区时始终出现纵向滚动条，不再出现内容被裁切但无滚动反馈的状态。

### 2. 收敛 Hotkeys 密度

让热键分组卡片、行内标签与录制器之间的关系更稳定，减少“长输入框感”，提升卡片之间和行之间的节奏感。

### 3. 精修 Select 组件

保留自定义 `SSelect`，但收紧尺寸、层级与信息密度，让它更像桌面设置控件而不是粗糙的表单下拉。

### 4. 理顺信息层级

选择器只负责“选择项”；终端路径等长信息退回详情区展示，避免同一控件同时承担“选择 + 详情”两种职责。

---

## 方案对比

### A. 保守修补

只补滚动条、修少量间距和文案，不调整控件比例与布局。

优点：
- 风险最低
- 修改量最小

缺点：
- 很难真正解决 Hotkeys 过长和 Select 粗糙的问题
- 无法达到参考图那种更有呼吸感的设置面板节奏

### B. 结构精修（推荐）

保留当前 Settings 窗口结构与分区逻辑，通过滚动容器、卡片节奏、Hotkeys 行布局和 Select 组件精修来统一界面。

优点：
- 直接命中本轮 4 个问题
- 风险可控，不动业务逻辑
- 能明显提升精致度

缺点：
- 需要同时修改多个 Settings 组件和样式文件

### C. 大幅重排

更激进地重做 General / Hotkeys 的信息架构，接近新的偏好面板范式。

优点：
- 视觉上限最高

缺点：
- 超出本轮“只修视觉呈现”的范围
- 回归风险明显放大

结论：采用 **B. 结构精修**。

## 详细设计

## 一、整体布局与滚动策略

### 设计

- 顶部 `SSegmentNav` 所在 topbar 固定，始终不参与内容滚动。
- Settings 主体仅保留一个纵向滚动容器，即 `.settings-content`。
- 为窗口根节点、内容容器、页面 section 建立完整的 `min-height: 0 / overflow-y: auto` 约束链，确保溢出时一定出现滚动条。
- 普通页面继续采用居中阅读宽度；`Commands` 页面允许更宽，但仍保留上限。
- section 与 card 的垂直间距提升一档，让内容从“连续堆叠”变成更可读的分组布局。

### 目的

- 解决“东西太多没有滚动条”。
- 让后续 Hotkeys 和 Select 精修拥有稳定的空间基础。

---

## 二、Hotkeys 卡片与行布局

### 设计

- 保留当前按分组展示快捷键的结构，不改字段定义和保存逻辑。
- 每个热键分组卡片统一卡片外间距、标题区节奏与行内 padding。
- 热键行改为稳定两栏：
  - 左侧为标签区，允许更自然换行。
  - 右侧为录制器区，宽度更克制，不再表现为超长输入框。
- 录制器区在常规宽度下采用固定上限宽度，在窄宽度下自动退化为上下布局，避免重叠和挤压。
- 冲突提示继续保留行内展示，但视觉附着于录制器，不再拉坏整张卡片的节奏。

### 目的

- 解决“快捷键设置太长”。
- 解决较窄窗口下的控件挤压、重叠与卡片缺乏呼吸感。

---

## 三、General 页 Select 与终端信息呈现

### 现状判断

当前 `SSelect` 为项目自定义组件，`SettingsGeneralSection.vue` 中默认终端选项把 `path` 作为 `description` 一并展示。对于桌面设置场景，这会造成：

- option 过高，视觉过重；
- 长路径稀释“选择”信息；
- 下拉面板宽度与密度失衡。

### 设计

- 继续使用自定义 `SSelect`，不切换到原生 `select`。
- 下拉触发器整体缩一档：高度、padding、边框厚度、圆角与箭头占比统一收紧。
- 下拉列表项默认只展示 `label`，不展示终端路径 description。
- 默认终端下拉框只显示终端名称，例如 `PowerShell`、`Windows Terminal`。
- 终端路径保留在下方独立只读行，用单行截断方式展示。
- 语言等其他 Select 同步采用同一套更紧凑的尺寸规范，保证 Settings 内部控件统一。
- 如果后续仍需展示额外信息，优先走 hover title 或详情区，而不是重新塞进 option description。

### 目的

- 解决“下拉框写得太潦草”“下拉框太大”“终端选择不需要展示路径”。
- 理顺“选择器”与“详情展示”的职责边界。

---

## 四、间距、密度与视觉基调

### 设计

- 保持 ZapCmd 当前深色主题体系，不引入新的高饱和色方向。
- 卡片外间距增大，卡片标题与内容之间的距离拉开。
- 行高与控件高度统一收紧一档，变得紧凑但不压抑。
- 标签、hint、mono path、select value 之间建立清晰的视觉主次。
- hover / focus / border / background 反馈保持克制，以桌面设置页的稳定感为优先。

### 参考原则

- 参考 `docs/bug_img/gemini.png` 的布局呼吸感与控件比例。
- 不复制 Gemini 的完整视觉语言，只借鉴其层级关系和密度控制。

## 影响文件

- `src/styles/settings.css`
  - 修正高度链、滚动容器、卡片间距、General / Hotkeys 的节奏与响应式布局。
- `src/components/settings/parts/SettingsHotkeysSection.vue`
  - 调整热键行 label / recorder 结构与布局约束。
- `src/components/settings/parts/SettingsGeneralSection.vue`
  - 默认终端选项改为仅展示 label，路径继续作为独立只读行。
- `src/components/settings/ui/SSelect.vue`
  - 精修 trigger / panel / option 的尺寸与层级，减少粗糙感。

必要时可补充相关 layout tests，但本轮不预设改动 store、composable 或 Rust 窗口逻辑。

---

## 非目标

- 不修改 settings 数据结构、保存逻辑或即时保存策略。
- 不新增设置项，不重命名现有设置项。
- 不重做 topbar / 原生标题栏结构。
- 不把 Settings 重新设计成全新产品风格。

---

## 风险与注意点

- `SSelect` 为 Teleport 到 `body` 的自定义组件，精修尺寸时要注意 panel 定位、滚动与焦点行为不能回归。
- Hotkeys 布局收紧后，必须保证长文案与冲突提示在中窄宽度下依然可读。
- 滚动修复应优先通过 CSS 高度链完成，避免引入额外 JS 滚动逻辑。

---

## 验收标准

1. Settings 内容超出窗口高度时，内容区稳定出现纵向滚动条，顶部 Tab 保持固定。
2. Hotkeys 页面不再出现录制器过长、标签挤压或控件重叠问题。
3. Hotkeys 分组卡片之间、卡片内部各行之间的呼吸感明显改善。
4. `SSelect` 触发器与下拉面板整体更紧凑，视觉上与当前深色设置页一致。
5. 默认终端下拉列表不再展示路径；路径继续保留在独立只读行中，并支持单行截断。
6. General 页面中语言等其他 Select 也同步采用统一的紧凑尺寸规范。
7. 不引入业务逻辑回归；若有相关测试，需要维持通过。
