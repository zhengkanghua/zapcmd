# Settings 窗口商业化审查设计文档

日期: 2026-03-19
方案: B — 商业化审查 Spec
状态: 设计已确认，待进入 implementation plan

---

## 背景

`settings` 当前已经完成“原生标题栏 + 应用 topbar + 独立内容区”的稳定版收口，并具备独立入口、模块化 store / composable、主题变量与 focused tests。  
但 Windows 实机反馈仍暴露两个直接问题：

1. 拖拽不跟手，存在明显延迟感。
2. 窗口打开时存在白色频闪。

在此基础上，本轮不直接进入代码修改，而是先输出一份“商业化审查 Spec”，用于回答以下问题：

1. 当前 `settings` 窗口的设计、架构、性能是否已经达到商业化做法。
2. 两个已测问题更可能来自哪里。
3. 后续修复应该按什么优先级推进。
4. 什么结果才算“修好了”，避免后续只做主观微调。

---

## 审查范围

本审查只覆盖独立 `settings` 窗口，不扩散到 launcher 主窗口或其他 overlay。

覆盖范围：

- Tauri settings 窗口创建与显示策略
- `settings.html` 首帧加载路径
- `src/main-settings.ts` / `src/AppSettings.vue` 组合根
- `src/components/settings/*` 壳体与各子页面
- `src/composables/settings/*` 的状态、持久化与命令页视图逻辑
- `src/styles/settings.css` / `tokens.css` / 主题变量

不在本轮范围：

- 新增设置项
- 合并 settings 与 launcher
- 重做整套品牌视觉系统
- 讨论移动端 / Web standalone 适配

---

## 已确认输入

### 用户实测问题

1. 拖拽不跟鼠标，存在延迟。
2. 窗口打开有白色频闪。

### 代码侧已定位事实

1. settings 窗口由 Rust 侧 `open_or_focus_settings_window` 创建，构建后立即 `show()` + `set_focus()`。
2. `settings.html` 在 Vue 挂载前只同步 `data-theme` / `data-blur`，没有直接给首帧上深色底。
3. 全局 reset 把 `html/body/#app` 设为 `background: transparent`。
4. 命令页搜索与筛选通过 `commandView` 即时更新，并在根组件中立即持久化到 store / localStorage / BroadcastChannel。
5. `settings` 使用了自绘 topbar，但真实拖拽能力仍来自原生窗口标题栏，而不是 WebView 内部 drag-region。

## 目标

本审查文档要先给出“商业化”口径，而不是直接给修复 patch。

本轮目标：

1. 定义当前窗口距离“商业化体验”还差哪些关键项。
2. 给出有证据支撑的根因判断和置信度。
3. 把问题分成 P0 / P1 / P2，避免后续修复顺序失焦。
4. 产出后续 implementation plan 的上游输入。

## 方案对比

### A. 轻量问题单

只写问题、影响和建议。

优点：

- 输出快
- 适合临时沟通

缺点：

- 不能稳定承接后续 implementation plan
- 无法明确“商业化标准”和验收口径

### B. 商业化审查 Spec（推荐）

把现状、代码证据、根因判断、商业化标准、修复原则和验收标准一次性写清。

优点：

- 最适合当前阶段
- 能直接成为后续 plan 的上游
- 既能回答“是不是商业化”，又能约束“怎么修”

缺点：

- 比普通问题单更重

### C. 审查 + 实施一体长文

一份文档里同时包含审查结论和任务拆解。

优点：

- 信息全

缺点：

- 过重
- 后续维护成本高
- 审查结论与执行步骤容易互相污染

结论：采用 **B. 商业化审查 Spec**。

## 商业化标准定义

本项目中的“商业化做法”，不等于视觉花哨，而是满足以下底线：

### 1. 首帧稳定

- 打开窗口时不出现白底、闪底、主题延后切换。
- 用户看到的第一帧应该已经处于正确主题和正确背景层级。

### 2. 拖拽模型一致

- 用户看到哪里像标题栏，哪里就应该能拖。
- 不允许“视觉上的标题栏”和“真正可拖区域”分离。
- 拖拽应跟手，没有明显延迟、抽动或拖拽区域误判。

### 3. 状态分层清晰

- 需要跨窗口共享的设置才持久化和广播。
- 命令页搜索、筛选、临时 UI 状态应优先本地瞬态，不应每击键写入持久层。

### 4. 性能有余量

- 常见操作不应把列表全量重算、全量写存储、全量广播绑在一起。
- 复杂页面应具备可扩展空间，而不是“当前数据量勉强能跑”。

### 5. 视觉系统克制且一致

- 图标、字体、控件密度、hover 反馈、玻璃效果和层级要统一。
- 桌面设置页应优先稳、清晰、可信，而不是靠堆效果制造高级感。

## 当前实现简述

### 架构现状

- Tauri 侧窗口构建集中在 `src-tauri/src/windowing.rs`。
- Web 入口使用独立 `settings.html` + `src/main-settings.ts`。
- 组合根 `src/AppSettings.vue` 负责 theme、i18n、settingsStore、命令页聚合和窗口事件。
- `useSettingsWindow` 已拆成 `hotkey / general / persistence / route / terminal / viewModel` 多模块。
- `SettingsWindow.vue` 基本保持“壳体 + route 渲染”职责。

### 设计现状

- 当前是深色稳定版：原生标题栏 + 应用 topbar + bounded content width。
- 普通路由最大宽度 720，Commands 最大宽度 1120。
- 有主题 token 体系，但目前只有单主题 `obsidian`。

### 性能现状

- Hotkeys / General / Appearance / About 规模可控。
- Commands 路由已具备搜索、筛选、分组和 sticky toolbar，但仍走全量映射、全量 filter/sort、全量渲染路径。

## 审查结论

### 一、总体判断

当前 `settings` 窗口 **不是 demo 级实现**，已经具备中上水平的工程化基础。  
但它 **还不能直接算商业化完成态**，原因不在“有没有组件拆分”，而在：

1. 首帧体验仍有明显 bug。
2. 窗口拖拽策略和视觉暗示不一致。
3. 命令页把“瞬态视图状态”和“持久化设置状态”混在一起。
4. 视觉系统仍停留在“能看、统一”，还没到“可信、成熟、可发布”的品牌完成度。

换句话说：  
**架构基础已经接近商业项目，但窗口壳体体验和性能边界还没有收口到商业化水平。**

---

### 二、按严重度排序的问题

### P0：首帧白色频闪

#### 结论

这是当前最明确、最不应该出现在商业桌面应用中的问题。

#### 证据链

1. Rust 侧创建窗口时先 `visible(false)`，但 build 完成后立即 `show()`。
2. `settings.html` 首帧脚本只设置 `dataset`，没有直接设置深色 `background-color`。
3. reset 层把 `html/body/#app` 全部设成透明。
4. 当前窗口又不是 `transparent: true` 的自绘窗体，而是普通原生标题栏窗口。

#### 判断

高置信度认为：  
白闪来自 **窗口已经显示，但 WebView 首帧背景仍是系统默认底色/白底，随后才被 CSS 深色主题覆盖**。

#### 商业化评语

商业化桌面应用不应接受“偶发白闪”。  
这是应优先处理的 P0 体验缺陷。

---

### P1：命令页搜索/筛选状态被错误持久化

#### 结论

这是当前最值得修的性能与架构问题，比单纯调 CSS 更关键。

#### 现象

- 命令页搜索框 `@input` 即时更新 `commandView.query`。
- 根组件收到更新后立即 `persistImmediate()`。
- `persistImmediate()` 会写入 settings snapshot，并广播跨窗口同步。

#### 判断

这意味着“搜索词”“筛选器”等本应是页面临时态的内容，被当成全局持久设置处理。  
在数据量变大、输入更快或系统 IO 紧张时，会放大为：

1. 输入卡顿
2. 无意义存储写入
3. BroadcastChannel 噪声
4. 其他窗口被动 reload settings

#### 商业化评语

商业化做法应明确区分：

- **用户偏好**：需要持久化
- **当前视图态**：优先瞬态或 debounce 持久化

当前实现把两者混在一起，属于“工程结构尚可，但状态分层不成熟”。

---

### P1：拖拽策略与视觉壳体不一致

#### 结论

这是当前“拖拽不跟手”的第一嫌疑项。

#### 现状

- settings 使用原生标题栏。
- WebView 内又绘制了一条很像标题栏的 `.settings-window-topbar`。
- topbar 本身没有 `data-tauri-drag-region`。

#### 判断

用户很容易把 topbar 误判为可拖区域，但真正的拖拽仍然是系统标题栏在负责。  
这会造成两种问题：

1. 用户抓错区域，以为“拖不动/有延迟”。
2. 即便抓的是系统标题栏，视觉上也会产生“不是我正在拖的那条”的认知错位。

如果 Windows 实机确认“抓住系统标题栏本身也会迟滞”，那还需要继续查毛玻璃、合成器、窗口显示时机或系统渲染负担。  
但在进入更深排查前，**壳体策略不一致** 已经足够被判定为设计问题。

#### 商业化评语

商业化产品不会让“看起来像标题栏”和“真正标题栏”分成两层语义。  
当前方案更像“稳定版过渡态”，不是最终商业化方案。

---

### P2：命令页缺少性能余量

#### 结论

当前 Commands 页能用，但还不是面向更大数据量的商业做法。

#### 现状

- 全量 map 为 `CommandManagementRow[]`
- 全量 filter
- 全量 sort
- 全量 `v-for` 渲染

#### 判断

在现在的数据规模下问题可能不明显，但从设计上看，它更接近“内部后台列表页”，不是“有余量的桌面设置页”。

#### 商业化评语

短期内未必需要直接上虚拟列表，但至少要做到：

1. 先拆掉每击键持久化
2. 避免不必要的全树重算
3. 在必要时给列表渲染提供 memo/虚拟化空间

---

### P2：视觉完成度仍偏工程态

#### 结论

视觉已经统一，但还不够“成品化”。

#### 具体表现

1. 顶部导航仍使用字符图标，而不是统一 SVG 图标体系。
2. 字体仍是系统默认栈，稳定但品牌感弱。
3. 玻璃、hover、阴影和 sticky toolbar 的视觉效果略重，容易在 Windows 上带来“发肉”的体感。
4. 单主题虽然干净，但说明当前更像“主题系统已落地，品牌系统未完成”。

#### 商业化评语

现在的 UI 已经高于纯工具感，但还没到成熟品牌产品的完成度。  
这项优先级低于白闪、拖拽和命令页状态分层。

---

### 三、架构评价

### 做得好的部分

1. 入口清晰：settings 已独立为多入口，而不是塞在 launcher 内。
2. 组合根职责基本合理：`AppSettings.vue` 负责接线，`SettingsWindow.vue` 负责壳体。
3. settings 相关逻辑已经按领域拆分，而不是堆在单文件。
4. token / theme / styles 有统一语义层，不是散落硬编码。

### 当前不足

1. `AppSettings.vue` 仍然承担了较多聚合职责，尤其是命令页持久化策略把 view-state 问题也带进来了。
2. settingsStore 同时容纳“真正设置”和“命令页界面状态”，边界不够纯。
3. 跨窗口同步策略过粗，缺少对“是否值得广播”的分层。

### 结论

架构上是 **商业项目中期状态**，不是架构灾难。  
真正的问题不是“拆得不够”，而是 **状态边界还有一层没有收口完**。

---

### 四、根因置信度

### 高置信

1. 白闪由首帧底色与显示时机造成。
2. 命令页即时持久化会放大输入卡顿和无效 I/O。

### 中置信

1. 拖拽迟滞主要来自原生标题栏与 WebView topbar 的视觉/行为错位。
2. Windows 上额外的 blur / sticky / shadow 效果会放大“发肉”体感。

### 待 Windows GUI 手验确认

1. 仅抓住系统标题栏时，是否仍存在明显迟滞。
2. 白闪是否 100% 发生在首次打开，还是每次打开都可见。
3. Commands 页在较大命令集下的输入体感是否已可见退化。

---

## 修复原则

### 1. 首帧优先于所有视觉润色

先解决白闪，再谈其他 polish。  
首帧不稳定时，任何视觉精修都不成立。

### 2. 先分离瞬态状态与持久状态

优先把 `commandView` 中的搜索/筛选视图态从“每击键持久化”中剥离。  
这是后续性能优化的第一步。

### 3. 只保留一种拖拽语义

必须二选一：

1. 完全拥抱原生标题栏，不再让 topbar 长得像可拖壳体。
2. 真正做自定义 titlebar/drag strip，并为 Windows 做明确 drag-region 验收。

### 4. 特效克制

Windows 桌面设置页优先稳态、跟手和清晰。  
凡是会干扰拖拽、滚动和首帧稳定的 blur / shadow / transform，都应让位于体感。

### 5. 商业化不等于大改版

本轮优先修壳体体验与性能边界，不做大范围视觉重设计。

### P0

- 消除 settings 打开白闪

### P1

- 重构 Commands 视图态持久化策略
- 统一拖拽模型与视觉壳体

### P2

- 为 Commands 列表建立性能余量
- 收口图标、字体、hover、玻璃效果的商业化完成度

## 验收标准

### 首帧

1. Windows 下连续打开 settings 10 次，不出现可见白色首帧。
2. 首帧看到的背景、主题和 topbar 层级稳定一致。

### 拖拽

1. 用户可感知的“标题栏区域”与真实拖拽区域一致。
2. 连续拖动窗口时，窗口跟手，无明显迟滞和抽动。
3. 不出现“某条看起来像标题栏但拖不动”的区域。

### 性能

1. Commands 搜索输入不再每击键写入持久层。
2. 输入搜索词时无明显卡顿。
3. 切换筛选器不会触发无意义跨窗口广播。

### 架构

1. 真正需要跨窗口同步的设置仍然保留同步。
2. 临时视图状态与持久设置状态有明确边界。

### 视觉

1. 首帧、拖拽、滚动和 sticky toolbar 的体感优先于特效。
2. 图标、字体、控件和状态反馈保持统一，不再出现“工程态字符图标”破坏成品感。

## 后续 implementation plan 边界建议

下一阶段 implementation plan 建议只覆盖以下内容：

1. P0 白闪修复
2. P1 Commands 视图态持久化策略重构
3. P1 拖拽模型与 topbar 壳体统一

不建议在同一波次内同时做：

- 新主题扩展
- 全量品牌视觉升级
- settings 与 launcher 的结构合并

原因：  
这三项会把“商业化修复”重新拉回“设计大工程”，不利于快速闭环。

## 下一阶段建议

下一步进入 `writing-plans`，产出一个聚焦 P0 / P1 的实施计划，建议结构如下：

1. 先锁定首帧与持久化边界的 contract / regression tests

本 Spec 的作用不是直接改代码，而是先把“当前离商业化还差什么”写死，避免后续修复只凭感觉推进。
