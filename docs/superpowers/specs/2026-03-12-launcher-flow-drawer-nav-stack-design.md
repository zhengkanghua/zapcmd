# ZapCmd｜Launcher 左侧 Flow 抽屉（参数/高危）+ 右侧 Review 抽屉并存（左进右出）设计稿

日期：2026-03-12  
状态：已与需求方对齐（待实现）  
关联：`docs/superpowers/specs/2026-03-10-launcher-review-drawer-overlay-design.md`（右侧 Review 抽屉基线）

## 1. 背景与问题

当前 Launcher 存在两类“弹窗式”交互：

1) **命令参数填写**：点击/热键触发后出现居中弹窗（Param Overlay）。  
2) **高危命令确认**：执行前出现居中确认弹窗（Safety Overlay）。

在桌面应用场景下，这类“阻塞式弹窗”会破坏连续操作体验，也与项目“执行流/暂存区抽屉”既有交互心智不一致。

本设计稿目标是把上述弹窗统一替换为**抽屉式导航栈**：  
参数填写/高危确认走**左侧 Flow 抽屉**，并与既有**右侧 Review 抽屉**并存。

## 2. 目标（What）

### 2.1 形态目标

- 参数填写、高危确认不再使用弹窗（modal）。
- 引入 **左侧 Flow 抽屉**（导航栈 / navigation stack）：
  - 从**左侧进入**供用户填写/选择。
  - **取消**：抽屉**向左退回**并关闭。
  - **确认**（无论“立即执行”还是“加入执行流”）：抽屉**向右退场**并关闭，给用户“从左进入、向右完成”的视觉动线。
- 保留并复用 **右侧 Review 抽屉**（队列/暂存区），并允许在 Flow 抽屉开启期间随时打开/关闭。

### 2.2 关键布局目标（宽度规则）

抽屉都挂载在**搜索框下方内容区**（不覆盖 search capsule），并遵守以下宽度规则：

1) **只有一个抽屉打开**（左 Flow 或右 Review 任意一个）：
   - 抽屉宽度 = 内容区宽度的 **2/3**（沿用 Review 抽屉基线口径）。
2) **左右抽屉同时打开**（Flow + Review）：
   - 自动均分：左 1/2 + 右 1/2（各自可交互）。

> 说明：宽度从 2/3 ↔ 1/2 的切换需要有平滑过渡动画（或在 reduce motion 下无动画）。

### 2.3 交互目标（并存能力）

- 用户在填写参数/高危确认时，仍可能通过热键或点击入口**打开执行流（Review 抽屉）**。
- 当触发新的抽屉出现时：
  - 若当前仅有单抽屉（2/3）→ 进入双抽屉（1/2+1/2）。
  - 关闭其中一个 → 另一个回到单抽屉（2/3）。

## 3. 非目标（Out of Scope）

- 不改变命令模板、分类、执行逻辑本身（仅改 UI 呈现与交互衔接）。
- 不在本稿中处理 Settings Window 的 `window.confirm`（另见“补充发现”）。

## 4. 术语与区域定义

- **Search Capsule**：顶部搜索条区域（输入框 + 队列入口等）。
- **内容区（Content Area）**：Search Capsule 下方的区域（结果抽屉 / 空态 / floor 占位）。
- **Review 抽屉**：右侧队列/暂存区（现有 Review Overlay）。
- **Flow 抽屉**：左侧导航栈容器（本设计新增），承载 Param/Safety 等流程页。

## 5. 信息架构（IA）

### 5.1 Flow 抽屉导航栈（左侧）

Flow 抽屉以“导航栈”方式在同一抽屉内切换页面：

- `Param`：参数填写页（表单）
- `SafetySingle`：单命令高危确认页
- `SafetyQueue`：队列执行高危确认页

导航栈基本规则：

- **从 Param 进入 Safety**：不关闭 Flow 抽屉，而是在抽屉内部前进到 Safety 页（避免“关一个再开一个”的弹窗感）。
- **Safety 的取消行为（已确认）**：当 Safety 来源于 Param 流程时，点击“取消”回到 Param（保留已填写参数）。

### 5.2 Review 抽屉（右侧）

Review 抽屉继续采用现有 “面板内 2/3 覆盖” 基线，但在“左右并存”模式下需要支持变为 1/2。

## 6. 布局与层级（Layout）

### 6.1 抽屉挂载位置

两类抽屉都必须挂载在 Launcher 主面板的**内容区**内：

- 抽屉覆盖内容区，不覆盖 Search Capsule。
- 打开抽屉时，内容区底层结果保持可见但不可交互（`inert`/`aria-hidden` 或等价方案）。

### 6.2 单抽屉模式（2/3）

- **仅 Flow 抽屉打开**：左对齐，宽度 2/3；右侧 1/3 作为“上下文背景”区域。
- **仅 Review 抽屉打开**：右对齐，宽度 2/3；左侧 1/3 作为“上下文背景”区域。
- 单抽屉模式下保留“点击背景关闭”的交互（背景 scrim 仅覆盖内容区，不覆盖 Search Capsule）。

### 6.3 双抽屉模式（1/2 + 1/2）

- 左右抽屉同时存在时：
  - 两边各占 1/2。
  - 两边都可交互（不再存在可点击关闭的“空白 scrim 区域”）。
  - 关闭依赖：显式关闭按钮 / Esc（见交互契约）。
- 可选：两抽屉中间提供 1px 分隔线或弱对比边界以提升层次感（保持克制）。

## 7. 动效与过渡（Motion）

### 7.1 左侧 Flow 抽屉（左进右出）

- 打开：从左侧滑入（`transform: translateX(-100%) → 0`）。
- 取消：滑回左侧并关闭（`0 → -100%`）。
- 确认：继续向右滑出并关闭（`0 → +100%` 或等价的“跨越内容区向右退场”）。

### 7.2 右侧 Review 抽屉

沿用既有抽屉动效（右进右出），但需要支持：

- 单抽屉（2/3）↔ 双抽屉（1/2）宽度切换时的平滑过渡。

### 7.3 减弱动效

遵循 `prefers-reduced-motion: reduce`：

- 关闭所有滑动/淡入动画，直接切换显示状态。

## 8. 交互契约（Behavior）

### 8.1 打开 Flow 抽屉（参数填写）

触发：用户执行/加入队列一个“需要参数”的命令（现有语义不变）。

行为：
- 打开 Flow 抽屉进入 `Param` 页。
- `Param` 页提交：
  - 必填缺失：留在 `Param` 页并给出错误反馈（沿用既有反馈机制）。
  - “加入执行流”：成功后触发 Flow 抽屉**向右退场**并关闭。
  - “立即执行”：
    - 若命令命中高危确认：在 Flow 抽屉内前进到 `SafetySingle`（不退场）。
    - 若无高危确认：触发 Flow 抽屉**向右退场**并关闭，然后执行。

### 8.2 高危确认（Safety 页）

#### 8.2.1 SafetySingle（来源：Param → Execute）

- 取消（已确认）：回退到 `Param`，保留已填参数。
- 确认：Flow 抽屉**向右退场**并关闭，然后执行命令。

#### 8.2.2 SafetyQueue（来源：Queue Execute）

- 取消：关闭 Safety（Flow 抽屉按“取消”语义向左退场）。
- 确认：Flow 抽屉向右退场并关闭，然后执行队列。

### 8.3 Review 抽屉与并存

- Flow 抽屉打开期间，用户仍可触发打开/关闭 Review 抽屉（入口与热键语义保持一致）。
- 当 Review 抽屉在 Flow 期间打开：
  - 自动进入双抽屉模式（1/2 + 1/2）。
  - 最近打开的抽屉获得焦点（见可达性）。
- 当双抽屉模式下关闭一侧抽屉：
  - 另一侧回到单抽屉模式（2/3）。

### 8.4 Esc 行为（优先级）

沿用现有 Escape 优先级语义（从“更危险/更局部”到“更全局”）：

1) Safety（若在 Safety 页）  
2) Param（若在 Param 页）  
3) Review（若 Review 抽屉打开）  
4) 清空 query  
5) 隐藏主窗口

> 说明：双抽屉模式下按 Esc 将优先退出 Flow（Safety/Param），再退出 Review。

## 9. 可达性（A11y）与焦点管理

### 9.1 单一对话语义

为避免“同时存在两个 aria-modal dialog”的可达性陷阱，建议以**内容区 Drawer Host**作为唯一 `role="dialog"` / `aria-modal="true"` 的语义容器：

- Drawer Host 打开时：底层结果区 `inert + aria-hidden`。
- Drawer Host 内部包含：
  - 左侧 Flow 抽屉 panel（`section`）
  - 右侧 Review 抽屉 panel（`section`）

### 9.2 焦点规则

- 打开 Flow：默认聚焦到 Param 的第一个输入框；若为 Safety，则聚焦到“取消”或明确的首要操作按钮（防误触）。
- 打开 Review：沿用既有规则（优先聚焦 active card，否则聚焦关闭按钮）。
- 双抽屉：最后打开的一侧获得焦点；Tab 顺序限制在 Drawer Host 内（可采用统一 focus-trap）。

### 9.3 键盘确认安全性

高危确认页不应在“焦点位于另一侧抽屉”时被全局 Enter 误触确认。建议把“Enter 确认”限定在：

- 焦点位于 Safety 页内，且不在输入控件中；
- 或者仅当“确认按钮”处于 focus 时触发。

（实现阶段再结合现有热键测试用例微调，避免破坏既有键盘流。）

## 10. 验收标准（UAT / Done）

### 10.1 参数填写（Flow / Param）

1. 触发需要参数的命令时不出现居中弹窗；出现左侧 Flow 抽屉（内容区内，不覆盖 Search Capsule）。
2. 取消：抽屉向左退场并关闭；焦点回到搜索输入框。
3. 确认加入队列：抽屉向右退场并关闭；队列计数更新；Review 状态不被强制改变。
4. Flow 打开期间可打开 Review；两抽屉均分；两边可交互。

### 10.2 高危确认（Flow / Safety）

1. 高危确认不再以居中弹窗出现；显示为 Flow 抽屉内的 Safety 页。
2. 从 Param 进入 Safety 后点击“取消”回到 Param（保留参数值）。
3. 点击“确认执行”：Flow 抽屉向右退场并关闭，然后执行。

### 10.3 布局与动效

1. 单抽屉：宽度为内容区 2/3；双抽屉：均分 1/2 + 1/2。
2. 2/3 ↔ 1/2 切换有平滑过渡；`prefers-reduced-motion` 下无动画。

## 11. 补充发现（弹窗清点结果）

除 Launcher 的 Param/Safety 弹窗外，项目还有一处原生弹窗：

- Settings 关闭前未保存确认：`src/composables/settings/useSettingsWindow/persistence.ts` 使用 `window.confirm(...)`

建议另起一份 Settings 交互设计稿，用“路由/导航栈内确认页”替代该 confirm（避免阻塞式弹窗）。
