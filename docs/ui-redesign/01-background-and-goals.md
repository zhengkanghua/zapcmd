# 01. Background and Goals

## 背景

ZapCmd 当前已经具备明确的产品雏形：

- 全局热键唤起
- 搜索命令
- 支持参数填写
- 支持暂存/队列执行
- 独立 `settings` 窗口

从功能完整度看，它已经不是 Demo，而是一个可以使用的桌面工具。

但从视觉与产品成熟度看，**主窗口** 仍停留在“工程可用版”：

- 主窗口信息层级成立，但视觉系统不稳定
- 暂存区存在，但与主搜索形成双焦点竞争
- 背景透明度过高，导致产品气质被壁纸劫持
- 当前绿色被同时用作品牌色、选中色、按钮色、成功色，语义冲突明显

因此，这次工作不是“再 polish 一下”，而是一次 **主窗口 UI shell 级别的大重构前置规划**。

## 范围收窄说明

### 本轮范围内
- 仅讨论 **launcher 主窗口**
- 仅讨论搜索态、Review 态、队列入口、长命令展示、主窗口热键与窗口尺寸

### 本轮范围外
- `settings` 保持 **独立窗口**
- 本轮不将 `settings` 并入主窗口
- 本轮不重构 `settings`
- 如后续要升级 `settings`，应单独开一个独立专题

## 本次工作目标

本轮只做 **文档先行**，目标是把以下问题说清楚：

1. 主窗口最终应该是什么形态
2. 暂存/队列能力应该如何呈现，而不破坏搜索效率
3. 新颜色系统和主窗口视觉系统应该是什么
4. 如果按推荐方案落地，会影响哪些主窗口代码模块
5. 如何给外部 Agent / Gemini Canvas 写 Prompt 出主窗口 Demo

## 本次非目标

以下内容 **不在当前文档阶段**：

- 不直接改 Vue/Tauri 代码
- 不修改 Rust 后端执行逻辑
- 不调整命令目录 schema
- 不引入新的业务能力（如收藏、历史、推荐、AI 命令解释）
- 不把 `settings` 并入 launcher
- 不扩张 `settings` 的业务范围

## 成功标准

当下面几点都成立时，说明文档阶段完成：

### 方向层
- 能明确选出 1 个主方案
- 团队对“暂存区是不是常驻”不再反复摇摆
- 团队对“settings 是否并入主窗口”不再产生歧义

### 设计层
- 有完整颜色系统，不再依赖旧绿色
- 有主窗口的字体、间距、圆角、阴影、透明度、动效统一规范
- 有主窗口 Search / Review 的组件层级规则

### 工程层
- 能准确指出会动到哪些主窗口模块
- 能区分“高风险动到交互契约”的模块和“只改视觉”的模块
- 能给未来执行阶段拆出可落地的波次

### 协作层
- 能把 Prompt 直接交给其他前端 Agent 生成 **主窗口 Demo**
- Demo 产出后，能与当前代码和文档一一对照复盘

## 核心设计原则

### 1. 单焦点优先
ZapCmd 主窗口的本质是命令启动器。
主窗口必须保证：用户进入时只会被一个主要工作区吸引。

### 2. 队列是高级能力，不是主舞台
暂存/队列很有价值，但它不应该抢搜索区的视觉和交互优先级。

### 3. 开发者工具不等于“黑底 + 绿色”
现代开发者工具更常见的是：低噪音、冷静、强层级、少装饰、语义清晰。

### 4. 品牌色与状态色分离
品牌色是品牌；成功/危险/警告是状态语义。
这两类颜色不能继续混用。

### 5. `settings` 与 launcher 分工明确
- launcher：瞬时、高频、命令工作流
- settings：独立、低频、配置工作流

本轮只做 launcher，不改变这条分工边界。
