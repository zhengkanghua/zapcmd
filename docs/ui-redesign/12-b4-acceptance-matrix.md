# 12. B4 Acceptance Matrix and Demo Review

> 本文定义 B4 方案的验收矩阵。
> 目标是：无论是外部 Demo 评审，还是未来真正进入代码实现，都有一套统一的主窗口过线标准。

## 范围说明

### 本文覆盖
- launcher 主窗口
- Search / Review 结构
- 主窗口热键与窗口尺寸逻辑

### 本文不覆盖
- `settings` 视觉重构验收
- `settings` 表单与导航体验验收
- `settings` 并入 launcher 的讨论

`settings` 保持独立窗口，不在本轮验收范围内。

## 一、Demo 评审矩阵

> 适用于 Gemini Canvas、其他前端 Agent、设计稿或高保真 Demo 评审。

## A. 结构正确性

| 检查项 | 通过标准 | 不通过信号 |
|--------|----------|------------|
| 搜索态单焦点 | 用户第一眼只被搜索与结果吸引 | 仍然看到两个同等级工作区 |
| Review 结构 | Review 是 overlay，不是常驻并列栏 | Review 仍像右侧永久第二列 |
| 背景保留 | 打开 Review 后，能看到搜索上下文 | 一打开就完全像切页 |
| 背景锁定 | 背景不可交互 | 背景还能点击、聚焦、滚动 |
| floor protection | 1 条结果时界面仍稳定自然 | Review 打开时塌陷、过矮、割裂 |

## B. 视觉正确性

| 检查项 | 通过标准 | 不通过信号 |
|--------|----------|------------|
| 品牌色 | 采用蓝紫系或已选新品牌色 | 仍然是绿色品牌主色 |
| 成功色 | 绿色只用于成功/启用语义 | 品牌态和成功态继续共用绿色 |
| 透明度 | 壁纸只做氛围，不喧宾夺主 | 产品看起来像贴在壁纸上的黑色半透明网页 |
| Review 宽度 | 长命令明显不再拥挤 | 仍然像旧窄侧栏 |
| 遮罩边界 | 遮罩只在内部 shell | 整窗被一层蒙住或暴露原生窗口边界 |

## C. 交互可信度

| 检查项 | 通过标准 | 不通过信号 |
|--------|----------|------------|
| Review 入口 | queue summary pill 或等价入口明确 | 用户不知如何进入 Review |
| 关闭语义 | 能看懂如何退出 Review | 没有清晰关闭动作 |
| 操作层级 | Review 内的主动作清晰 | 执行、删除、排序混乱 |
| 长命令策略 | 主列表摘要 + Review 详情逻辑成立 | Demo 里长命令仍铺满首页 |

## Demo 评分建议

每项按 1~5 分打分：

- 结构正确性：40%
- 视觉正确性：35%
- 交互可信度：25%

建议通过线：
- **总分 >= 80 / 100** 才进入代码阶段

## 二、手动验收矩阵（代码阶段）

## 1. 主窗口基础路径

### Case 1：空闲搜索态
- 打开主窗口
- 不输入查询
- 确认主窗口干净、稳定、无额外队列面板

### Case 2：单结果搜索态
- 输入一个只返回 1 条结果的关键词
- 确认窗口高度较矮但不破坏布局
- 确认队列入口若存在，不抢主视觉

### Case 3：多结果搜索态
- 输入一个返回 6~8 条结果的关键词
- 确认结果列表滚动、选中态、预览摘要都稳定

## 2. Review 路径

### Case 4：1 条结果时打开 Review
- 在只有 1 条搜索结果的情况下，打开 Review
- 确认左侧抽屉先补足到 floor height
- 确认不会出现假结果项
- 确认 Review 打开后，背景可见但不可交互

### Case 5：多结果时打开 Review
- 在 6~8 条结果时打开 Review
- 确认窗口与 overlay 过渡自然
- 确认 Review 宽度足够支撑长命令

### Case 6：Review 内部滚动
- 准备多条队列项
- 打开 Review
- 确认列表内部滚动，不要求窗口继续无限增高

## 3. 键盘路径

### Case 7：搜索态打开 Review
- 搜索态按 `toggleQueue`
- 确认 Review 打开

### Case 8：搜索态打开 Review 并聚焦列表
- 搜索态按 `switchFocus`
- 确认 Review 打开且焦点落在列表区

### Case 9：Review 内部焦点循环
- Review 打开后，按 `Tab / Shift+Tab`
- 确认焦点只在 Review 内循环，不回到背景

### Case 10：Review 关闭
- Review 打开后按 `Esc`
- 确认关闭的是 Review，而不是直接隐藏主窗

## 4. 阻断层路径

### Case 11：参数弹层优先级
- 在搜索态或 Review 态触发参数弹层
- 确认参数弹层优先级高于底层层级
- 确认 `Esc` 只关闭参数弹层

### Case 12：安全确认优先级
- 触发高风险执行
- 确认安全确认层优先级最高
- 确认 `Enter` / `Esc` 正常工作

## 5. 窗口与视觉边界路径

### Case 13：拖拽区存在性
- Review 打开前后都检查顶部拖拽区
- 确认拖拽区未被吞掉

### Case 14：遮罩边界
- 打开 Review
- 确认遮罩只作用于内部 shell
- 确认没有暴露整窗边界和难看的原生外壳感

## 三、自动化测试优先级

## P0：必须补齐

### Launcher 行为回归
- `toggleQueue` 在搜索态打开 Review
- `switchFocus` 在搜索态打开 Review 并聚焦列表
- `Esc` 在 Review 态关闭 Review
- `Tab` 在 Review 态内部循环
- 1 条结果时存在 `drawerFloorHeight` 逻辑，但无假结果项

### 顶层阻断层回归
- 参数弹层打开时，Review / 搜索层不处理按键
- 安全确认打开时，其他层不处理按键

## P1：应尽快补齐

### 布局 / 视觉相关
- Review 打开时新增 filler 高度断言
- Review 打开时背景锁定类名 / 状态断言
- 关闭 Review 后焦点恢复断言

## 四、明确的不通过条件

以下情况出现任意一项，视为 **不通过**：

### 结构类不通过
- Review 仍然是常驻并列右栏
- 打开 Review 后背景还能交互
- 1 条结果时 Review 打开明显塌陷

### 语义类不通过
- 使用假结果项补高度
- `Esc` 在 Review 态直接隐藏主窗
- `Tab` 在 Review 态把焦点送回背景

### 视觉类不通过
- 继续用绿色作为品牌主色
- 遮罩做成整窗遮罩
- Review 宽度仍不足以承载长命令
- 主窗口视觉仍像贴在壁纸上的半透明网页卡片

## 五、评审建议流程

建议每轮都按这个顺序评审：

1. 先看结构是否正确
2. 再看交互层级是否正确
3. 再看视觉是否高级
4. 最后看是否值得进入代码阶段

### 不要反过来
不要先看“好不好看”，再看结构。B4 的成败首先取决于结构和层级，而不是配色表面。

## 六、进入正式开发前的最终闸门

只有当以下条件都满足，才建议进入代码阶段：

- [ ] B4 Demo 评分 >= 80
- [ ] floor 逻辑（= 4 条结果高度 + 搜索框高度；搜索框高度以 `.search-form` 容器渲染高度为准）已被团队接受
- [ ] 内部 shell 遮罩原则已被确认
- [ ] 热键迁移表无明显争议
- [ ] 状态机文档与组件拆分文档可支撑开发拆任务
- [ ] 团队接受“先尝试动态 resize，抖动则回退”的工程策略
- [ ] 团队确认本轮只做主窗口，`settings` 继续保持独立窗口
